# 项目全流程扩展方案

## 1. 文档目的

当前系统已经具备“项目审批辅助”能力，核心覆盖项目列表、项目详情、材料归集、规则校验、LLM 审批结论生成、评审反馈沉淀等能力。

后续目标不是简单追加两个页面，而是把系统从“审批系统”扩展为“项目全流程辅助系统”，统一承载以下三类业务阶段：

- 立项
- 任务单
- 验收

本文档用于沉淀后续开发蓝图，后续只要补充具体接口字段、页面原型、状态规则，即可直接进入编码。

## 2. 当前项目现状

从现有工程看，系统主体已经形成以下结构：

- 后端入口集中在 `app/main.py`
- 审批相关能力集中在 `app/approvals/`
- 技能管理能力在 `app/skills/`
- 前端页面集中在 `frontend/src/pages/`
- 分阶段材料建议沉淀在 `materials/`
- 项目级说明文档保留在 `docs/`

当前实现的核心特点：

- 主业务对象默认还是“审批”
- 路由命名、目录命名、结果存储命名都偏向 approval
- 规则引擎已经具备“按分类加载规则并输出结论”的雏形
- 项目列表和远程接口能力已经具备复用基础

因此，后续扩展不建议继续堆叠“审批专用逻辑”，建议转为“流程类型驱动”的统一骨架。

## 3. 建议的总体目标

建议将系统升级为一个统一的项目流程工作台，围绕一个项目主对象，管理多个流程阶段单据。

推荐主线关系如下：

- 一个项目对应一个立项单
- 一个项目对应多个任务单
- 一个项目对应一次或多次验收单

推荐统一生命周期视角如下：

1. 立项阶段：回答“为什么做、做什么、值不值得做”
2. 任务单阶段：回答“谁做、做哪些任务、何时交付、资源如何投入”
3. 验收阶段：回答“是否完成、完成到什么程度、遗留什么问题、能否结项/上线/付款”

## 4. 核心设计原则

### 4.1 不再以 approval 作为唯一中心

后续系统中的“审批”只是流程中的一种处理动作，不应再作为全部目录、接口和数据结构的唯一中心词。

### 4.2 统一流程骨架，按阶段扩展差异字段

建议保留一套统一的流程框架：

- 流程类型
- 基础信息
- 表单数据
- 规则结果
- LLM 结果
- 人工反馈
- 状态流转

立项、任务单、验收只在“业务字段”和“规则集”上区分。

### 4.3 公共能力沉淀到共享层

以下能力应抽成公共模块，而不是在三个阶段内重复实现：

- 远程接口拉取
- 文档归一化
- 规则执行
- LLM 分析
- 结果存储
- 页面通用布局
- 状态展示和操作日志

### 4.4 先做最小闭环，再做深度细化

推荐先形成“能看、能校验、能输出结论”的最小闭环，再细化复杂状态和复杂审批流。

## 5. 业务模型建议

### 5.1 主实体建议

建议至少抽出以下实体：

#### Project

项目主对象，贯穿三个阶段。

建议字段：

- project_id
- project_code
- project_name
- domain
- department
- project_manager
- project_type
- project_category
- fixed_project
- project_status
- flow_status

#### WorkflowInstance

表示项目在某个流程阶段的一张业务单据。

建议字段：

- workflow_id
- project_id
- workflow_type
- workflow_title
- workflow_status
- source_system
- source_id
- created_by
- created_at
- updated_at
- submitted_at
- approved_at

其中 `workflow_type` 推荐枚举值：

- initiation
- task_order
- acceptance

#### WorkflowDocument

表示某流程单据归一化后的正文数据。

建议字段：

- workflow_id
- version
- raw_snapshot
- normalized_document
- document_source
- saved_at

#### WorkflowReviewResult

表示规则引擎或 LLM 的分析结果。

建议字段：

- workflow_id
- review_mode
- review_category
- decision
- score
- statistics
- findings
- rule_results
- generated_at
- run_dir

其中 `review_mode` 推荐枚举值：

- deterministic
- llm
- manual

#### WorkflowFeedback

表示人工修订意见、复核意见、补充材料意见。

建议字段：

- workflow_id
- feedback_type
- author
- content
- attachments
- created_at

### 5.2 三类流程建议的字段边界

#### 立项

重点关注：

- 项目背景
- 项目目标
- 项目方案
- 项目价值
- OKR
- 范围
- 架构评审
- 预算与费用构成
- 组织与资源
- 里程碑

这部分可以最大程度复用当前审批能力。

#### 任务单

重点关注：

- 所属项目
- 所属阶段或迭代
- 任务目标
- 任务范围
- 输出物
- 负责人
- 参与人
- 计划开始时间
- 计划结束时间
- 工时或人天
- 依赖关系
- 风险与阻塞
- 完成标准

任务单更偏执行管理，规则应更多偏完整性和计划合理性校验。

#### 验收

重点关注：

- 所属项目
- 所属任务单或阶段
- 验收类型
- 验收范围
- 实际交付物
- 验收结论
- 问题清单
- 缺陷级别
- 责任归属
- 整改计划
- 验收附件
- 是否允许上线/结项/付款

验收规则应更偏结果核验和闭环核验。

## 6. 状态流转建议

建议先统一状态框架，再允许每类流程扩展自定义状态。

推荐统一基础状态：

- draft
- pending_submit
- pending_review
- reviewing
- rejected
- supplement_required
- approved
- accepted
- archived

建议按流程类型定义典型终态：

- 立项：approved / archived
- 任务单：approved / archived
- 验收：accepted / archived

## 7. 后端重构建议

## 7.1 目录调整建议

建议逐步从当前目录过渡到如下结构：

```text
app/
  main.py
  core/
  workflows/
    models.py
    constants.py
    registry.py
    document_store.py
    result_store.py
    feedback_store.py
  workflows/initiation/
    service.py
    mapper.py
    rules.py
  workflows/task_order/
    service.py
    mapper.py
    rules.py
  workflows/acceptance/
    service.py
    mapper.py
    rules.py
  integrations/
    iwork_client.py
    ...
  llm/
    review_service.py
  skills/
```

说明：

- `app/approvals/` 不建议一次性删除
- 可先保留 `approvals`，再逐步迁移到 `workflows/initiation`
- 当前已有的 `iwork_client`、文档存储、反馈存储能力都适合抽到 workflow 公共层

### 7.2 服务层建议

建议形成统一服务接口，例如：

```python
get_workflow_document(workflow_type, project_id, workflow_id=None, refresh=False)
run_workflow_review(workflow_type, document, review_mode="deterministic")
save_workflow_feedback(workflow_type, workflow_id, feedback)
list_workflows(workflow_type, filters)
```

这样后端接口和前端页面都可以统一。

### 7.3 规则引擎建议

当前规则引擎已经能按 category 执行确定性检查。后续建议演化为：

- 公共规则引擎负责调度
- 各流程类型定义自己的规则映射
- 公共输出结构保持统一

统一输出建议如下：

```json
{
  "workflow_type": "initiation",
  "workflow_id": "WF-001",
  "decision": "approved",
  "score": 86,
  "statistics": {},
  "findings": [],
  "rule_results": [],
  "generated_at": "2026-03-20T10:00:00+08:00"
}
```

### 7.4 存储建议

当前运行结果主要围绕 `approval_runs`。建议后续调整为按流程类型分目录。

推荐目录示例：

```text
runtime/
  workflow_runs/
    initiation/
    task_order/
    acceptance/
  workflow_documents/
    initiation/
    task_order/
    acceptance/
  workflow_feedback/
    initiation/
    task_order/
    acceptance/
```

这样后续排查、回溯、归档都会清晰很多。

## 9. 前端页面规划

建议前端不要再单独围绕 `/approval` 扩展，而是调整成统一的工作台结构。

### 9.1 页面结构建议

- 项目总览页
- 项目工作台页
- 立项页
- 任务单页
- 验收页
- 规则与技能页
- 反馈与复核页
- 管理配置页

### 9.2 建议路由

- `/projects`
- `/project/:projectId`
- `/project/:projectId/initiation`
- `/project/:projectId/task-orders`
- `/project/:projectId/task-orders/:workflowId`
- `/project/:projectId/acceptance`
- `/project/:projectId/acceptance/:workflowId`
- `/review-feedback`
- `/skills`
- `/admin/config`

### 9.3 页面职责建议

#### 项目总览页

展示项目列表、筛选、状态、各流程进度。

#### 项目工作台页

作为某个项目的统一入口，展示：

- 项目基本信息
- 立项状态
- 任务单列表
- 验收列表
- 最新评审结果
- 最新人工反馈

#### 立项页

复用当前审批详情能力，重点展示材料、规则结果、LLM 结论、补充建议。

#### 任务单页

重点展示任务清单、计划、责任人、产出物、风险、完成标准、任务评估结论。

#### 验收页

重点展示交付结果、验收项、缺陷/遗留问题、整改建议、最终结论。

## 10. 数据归一化建议

### 10.1 raw snapshot

## 11. 规则设计建议

### 11.1 立项规则

沿用当前规则体系，后续重点做两件事：

- 把规则绑定从“审批分类”逐步过渡到“流程类型 + 业务分类”
- 对现有规则映射函数做模块化拆分

### 11.2 任务单规则

首批建议做的最小规则：

- 是否绑定项目
- 是否有明确任务目标
- 是否有负责人
- 是否有计划开始和结束时间
- 是否有交付物
- 是否有完成标准
- 是否有风险或依赖说明
- 工期是否异常

### 11.3 验收规则

首批建议做的最小规则：

- 是否绑定项目或任务单
- 是否有验收范围
- 是否有实际交付物
- 是否有验收结论
- 是否有问题清单
- 遗留问题是否有整改计划
- 是否满足上线/结项/付款条件

## 12. 开发顺序建议

建议按以下顺序推进。

### 阶段一：统一骨架

- 抽 workflow_type 常量
- 抽公共 document/result/feedback 存储
- 抽公共 review service
- 前端抽统一项目工作台入口

### 阶段二：迁移立项

- 将现有 approval 能力包装为 initiation workflow
- 保持现有页面可用
- 新旧接口并存一段时间

### 阶段三：接入任务单

- 接任务单数据源
- 做任务单详情页
- 做任务单规则引擎
- 做任务单反馈与复核

### 阶段四：接入验收

- 接验收数据源
- 做验收详情页
- 做验收规则引擎
- 做验收结论输出

### 阶段五：统一项目工作台

- 打通项目主视图
- 做跨阶段导航
- 做阶段进度总览
- 做多阶段结果联动

## 14. 需要优先确认的关键问题

在正式编码前，建议优先确认以下问题。

### 14.1 任务单与项目的关系

需要确认是：

- 一个项目多个任务单
- 一个项目单阶段一个任务单
- 一个任务单可关联多个项目

当前建议按“一个项目多个任务单”实现。

### 14.2 验收与任务单的关系

需要确认是：

- 验收只针对项目
- 验收只针对任务单
- 验收可以针对项目，也可以针对任务单

当前建议支持两种挂接方式，但实现时先选一种主路径。

### 14.3 是否允许本地补录

如果远程接口字段不全，需要确认是否允许：

- 本地补充录入
- 本地编辑后保存
- 仅只读展示远程数据

这个会直接影响前后端是否需要表单编辑能力。

### 14.4 是否保留当前审批页面

建议短期保留，避免一次性替换造成使用中断。

## 15. 建议的近期实施清单

如果按最稳妥的方式推进，建议近期实施顺序如下：

1. 先统一命名，从“审批系统”提升为“流程系统”
2. 先抽 workflow 公共结构，不立刻大改业务逻辑
3. 先把现有审批包装成 initiation 模块
4. 再接任务单数据和页面
5. 最后接验收并打通项目总工作台

## 16. 编码准备结论

基于当前代码基础，后续完全可以直接进入开发，但前提是优先补齐以下最小输入：

- 任务单接口样例
- 验收接口样例
- 三类流程的页面结构草图或字段清单
- 三类流程的状态流转说明
- 任务单和验收的最小规则集

只要这些信息具备，后续即可按本文档直接进行后端接口扩展、前端路由扩展、数据结构重构和规则引擎扩展。
