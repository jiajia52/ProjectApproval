# 单个项目审批信息与接口清单

## 1. 文档目的

这份文档用于说明当前系统在“读取单个真实项目并进行审批”时：

- 需要哪些项目信息
- 分别通过哪些接口获取
- 当前代码里如何把接口返回映射成审批文档
- 哪些映射是明确的，哪些映射目前只是推断，需你确认后再调整

当前实现涉及的主要代码：

- `app/approvals/iwork_client.py`
- `app/approvals/remote_project_mapper.py`
- `app/approvals/approval_engine.py`
- `app/main.py`

## 3. 当前使用的接口清单

## 3.1 项目列表接口

用于列表展示，也用于补充单个项目的摘要字段。

- 接口：`POST /projectEstablishment/queryProjectEstablishmentList`
- 作用：查询项目列表，获取项目基础概况
- 当前用途：
  - 项目列表页展示
  - 查询单个项目时补充 `project_summary`

当前实际能拿到的典型字段包括：

- `id`
- `projectName`
- `serialNo`
- `belongTeamName`
- `belongDepartmentName`
- `projectManagerName`
- `projectFeeTypeName`
- `projectLevelName`
- `proBudget`
- `proBudgetYear`
- `projectStatus`
- `flowStatus`
- `flowStatusName`
- `longTermFlag`

### 3.1.2 项目状态选项

- 接口：`GET /projectCenter/queryProjectStatusList`
- 作用：获取“项目状态”下拉可选项
- 说明：
  - 你给的文档里写的是 `queryProjectStatusLis`
  - 当前代码实际已按 `queryProjectStatusList` 调通
  - 如果后续确认正式接口名不同，需要统一调整

## 4. 当前接口到审批字段的映射关系

## 4.1 项目概况 `project_summary`

来源主要是项目列表接口 `queryProjectEstablishmentList`。

| 审批字段 | 当前来源字段 | 说明 |
| --- | --- | --- |
| 项目名称 | `projectName` | 明确 |
| 项目编码 | `projectCode` 或 `serialNo` | 当前多数使用 `serialNo` |
| 归属领域 | `domainName` 或 `belongTeamName` | 当前大多回退到 `belongTeamName` |
| 部门 | `departmentName` 或 `belongDepartmentName` | 部分项目可能为空 |
| 项目经理 | `managerName` 或 `projectManagerName` | 明确 |
| 项目类型 | `projectTypeName` 或 `projectFeeTypeName` | 当前存在混用 |
| 项目类别 | `projectCategoryName` 或 `projectFeeTypeName` | 当前存在混用 |
| 项目状态 | `projectStatusName` 或 `projectStatus` | 当前多数直接取 `projectStatus` |
| 流程状态 | `flowStatusDisplay` 或 `flowStatusName` 或 `flowStatus` | 明确 |
| 是否固定项目 | `fixedProject` 推断 | 目前不是稳定字段 |
| 项目等级 | `projectLevelName` | 明确 |
| 预算年度 | `proBudgetYear` 或 `proBudget` | 当前这里很可能不对，需要确认 |


## 4.2 项目内容 `project_content`

来源接口：

- `GET /projectUploading/list/{projectId}`

当前映射方式：

- 读取接口返回中的记录
- 根据记录文本里是否包含关键词，猜测属于以下哪个板块：
  - 背景
  - 目标
  - 方案
  - 全景图
  - 年度管理模型
- 再从记录里抽取：
  - `title / uploadTitle / fileName / name / moduleName`
  - `content / uploadContent / remark / description`
  - `images / imageList / fileList / attachments / files`

这部分当前是“关键词推断映射”，不是强字段映射，风险最高。

| 审批字段 | 当前接口 | 当前提取方式 | 风险 |
| --- | --- | --- | --- |
| 背景标题/内容/图片 | `projectUploading/list` | 按“背景”关键词猜测 | 高 |
| 目标标题/内容/图片 | `projectUploading/list` | 按“目标”关键词猜测 | 高 |
| 方案标题/内容/图片 | `projectUploading/list` | 按“方案”关键词猜测 | 高 |
| 业务全景图标题/内容/图片 | `projectUploading/list` | 按“全景”关键词猜测 | 高 |
| 年度管理模型标题/内容/图片 | `projectUploading/list` | 按“管理模型/年度”关键词猜测 | 高 |

## 5. 当前审批规则对字段的真实要求

审批引擎当前会按以下口径判断是否“材料完整”。

## 5.1 项目内容类

以下每个板块都至少要求：

- 标题非空
- 内容非空
- 图片非空

并且“内容”长度过短时也会判定为不通过。

适用板块：

- 项目背景
- 项目目标
- 项目方案
- 业务全景图
- 年度管理模型

## 5.2 OKR 类

要求非空：

- 关联产品链
- 项目目标
- 战队 OKR
- 关联产品
- 时间范围开始和结束
- KR 列表

## 5.3 范围类

要求非空：

- 业务流程
- 项目范围明细
- 微服务

当前“微应用”没有真实来源，所以这项大概率会出问题。

## 5.4 架构评审类

要求非空：

- 业务架构评审
- 数据架构评审
- 技术架构评审
- 安全架构评审

## 5.5 TAM 模型类

要求至少存在一条完整模型记录，且包含：

- 标题
- 现状
- 受益部门
- 3 年目标
- 测算依据

## 5.6 里程碑类

要求以下计划都具备开始和结束时间：

- 立项计划
- 合同计划
- 目标计划

## 5.7 组织架构类

要求：

- 至少存在成员
- 每个成员至少有姓名、角色

部分项目类别还会额外要求：

- 开发模式非空

## 5.8 预算类

要求每条预算项都具备：

- 名称
- 金额
- 预算科目
- 测算依据
- 采购方式

## 5.9 费用变化点类

如果判定为固定项目，则要求：

- 历史投入分析
- 变化原因

如果判定为非固定项目，则当前只要求：

- 变化原因


## 6. 当前明显不稳或待确认的地方

这部分建议你优先看。

### 6.1 项目内容五个板块

当前都来自同一个接口：

- `projectUploading/list/{projectId}`

但系统现在只是靠关键词猜“这条记录属于背景/目标/方案/全景/年度模型”。  
如果真实接口里本来就有明确模块字段，应该改成“按模块字段取值”，不要再猜。

### 6.2 项目类型 / 项目类别

当前 `projectTypeName` 和 `projectCategoryName` 都可能回退到 `projectFeeTypeName`。  
这说明当前口径并不清晰，需要你确认：

- 项目类型到底对应哪个远程字段
- 项目类别到底对应哪个远程字段

### 6.3 固定项目判断

当前存在两套口径：

- 列表摘要：根据 `fixedProject / isFixedProject / fixedFlag / longTermFlag / 名称关键词` 推断
- 变化点接口：根据 `isChange == "1"` 推断

这明显不稳，建议确认唯一来源字段。

### 6.4 预算年度

当前 `budget_year` 取的是：

- `proBudgetYear`
- 如果为空则回退 `proBudget`

从字段名看，这里很可能混淆了“预算年度”和“预算金额”。

### 6.5 微应用

当前没有真实接口来源，固定为空。

### 6.6 架构评审四项

当前直接从 `eamapAndSystem` 的顶层字段取：

- `businessArchitecture`
- `dataArchitecture`
- `technicalArchitecture`
- `securityArchitecture`

需要确认真实返回是否真是这样，还是藏在嵌套节点中。

### 6.7 项目价值

当前是把 `value/infoNoTam` 的整个 `data` 直接转成一段文本。  
如果审批要的是某几个固定字段，这里肯定要重做。


## 7. 建议你修改时重点确认的内容

建议你按下面这份清单逐项标注“对 / 不对 / 需替换字段 / 需替换接口”。

### 7.1 项目概况
